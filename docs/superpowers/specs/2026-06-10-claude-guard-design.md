# claude-guard 设计文档

日期：2026-06-10
状态：已确认，待实现

## 1. 项目目标

claude-guard 是一个包在 Claude Code 外层的「会话守护 + 编排 + 迁移」工具，用 Python 实现，带 PySide6 桌面 GUI。它解决使用 Claude Code 的三个痛点：

1. **关机后自动接着跑**：电脑关机/断电后，开机自动恢复之前托管的 Claude 会话，恢复完整上下文并自动续接未完成的任务。
2. **上下文不丢、不用重跑**：不重新开始任务，而是基于 Claude 原生的会话历史 `--resume` 接着原会话往下跑（Claude 本身会自动压缩超长上下文）。
3. **跨机器迁移接着跑**：在 A 机导出会话状态成一个文件，拷到家里的 B 机导入还原，接着 A 机的上下文继续跑。

### 核心认知

Claude Code 已经把每次会话的完整历史实时存在本地（`~/.claude/projects/<项目哈希>/<会话ID>.jsonl`），并支持 `claude --resume <会话ID>` 恢复完整上下文。因此本项目**不存储、不解析 Claude 的上下文内容**，只在外层做「记账 + 编排 + 引用打包」。这样 Claude 升级内部格式时本项目不受影响。

### 已确认的关键决策

- **使用场景**：交互式 / 半自动 / 无人值守三种都支持，核心是无人值守自动续跑。
- **存储策略**：本地优先，手动导出/导入迁移，不依赖任何网络或云账号。
- **恢复语义**：恢复完整上下文 + 自动发续接指令（恢复+自动续跑）。
- **权限处理**：GUI 可配置（完全跳过 / 允许清单 / 卡住通知我），有全局默认值，单会话可覆盖。
- **完成判定**：检测输出静止 + 确认回到等待输入状态。
- **GUI 形态**：PySide6 桌面窗口，可打包成 exe。
- **开机自启**：GUI 可配置开关，默认关闭；注册失败时明确提示原因和兜底方案。

## 2. 整体架构

分三层，每层职责单一、可独立测试。核心边界：**核心引擎与 GUI 完全解耦，引擎可脱离界面用 CLI 跑通全部逻辑**——这是降低 PTY 风险的根本手段。

### 第一层：核心引擎（无界面，纯 Python）

- **SessionRegistry（会话注册表）**：记账模块。记录每个被托管会话的元数据——工作目录、Claude 会话 ID、目标任务描述、权限模式、状态（运行中/已暂停/已完成/卡住/异常中断）、当前轮次、最后活动时间。存成本地 SQLite。
- **PtyHost（PTY 托管层）**：项目最硬的核心。用伪终端（Windows 经由 `pywinpty`）把 `claude` 进程包起来，能读取全部输出、能写入指令。唯一直接与 Claude 进程打交道的模块，刻意隔离以便测试。
- **IdleDetector（静止检测器）**：盯着 PtyHost 的输出流，判断「这一轮干完了 / 卡在权限询问了 / 还在忙」。
- **Supervisor（监督器）**：编排大脑。组合上面三者——拉起会话、监控静止、按权限策略自动应答、一轮完成后发下一条续接指令、更新注册表状态、执行各类刹车（轮次上限等）。

### 第二层：持久化与迁移

- **Snapshotter（快照器）**：把「注册表 + 相关 Claude 原生 `.jsonl` 文件 + 清单」打包成一个 zip 导出文件；在另一台机器解包还原，并处理路径差异。

### 第三层：GUI（PySide6 桌面窗口）

控制面板，通过核心引擎暴露的接口操作，不碰底层细节。显示会话列表、实时输出、启停按钮、每会话权限配置、自启开关、导出/导入按钮。GUI 逻辑保持薄。

## 3. 开机自启与恢复流程

### 托管会话（正常运行时）
用户不再自己开终端敲 `claude`，而是在 GUI 点「新建任务」：填工作目录、目标任务描述、选权限模式。Supervisor 经 PtyHost 在后台拉起 `claude`，记下生成的会话 ID。此后该会话被托管——目录、ID、目标、状态全在 SessionRegistry 有账。运行中状态变化持续写回 SQLite，**任何时刻断电账本都是最新的**。

### 关机/断电时
无需特殊处理。Claude 原生 `.jsonl` 实时落盘，SQLite 账本实时落盘。断电只是进程没了，数据都在。

### 开机后自动恢复
- 开机自启由 **GUI 可配置开关**控制，默认**关闭**，用户主动打开时才写入 Windows 启动项（当前用户启动项，不需管理员权限）。
- 注册启动项失败（被安全软件/杀毒拦截、权限不足）时**不静默失败**：GUI 明确弹提示，说明失败原因、可能是哪个安全软件拦的、解决办法（在安全软件中信任 claude-guard，或手动把快捷方式放到 `shell:startup`）。
- 提供「检测自启状态」功能，GUI 能告知当前自启是否真正生效。

开机后（自启开启时）claude-guard 静默拉起核心引擎，引擎读 SQLite，找出上次状态为「运行中/暂停」的会话，对每个：
1. 用 PtyHost 重新拉起 `claude --resume <会话ID>`，恢复完整上下文。
2. 等 Claude 加载完、回到等待输入状态。
3. 按该会话目标任务，自动发一句续接指令（如「继续之前未完成的任务：<目标描述>」）。
4. 交给 Supervisor 接管，进入「静止检测 → 自动应答 → 发下一条」循环。

### GUI 在恢复中的角色
GUI 可选。引擎可无界面后台恢复并续跑；用户想看进度时打开 GUI，它连上已在跑的引擎，显示实时输出和状态。

### 取舍
恢复后的续接指令是自动发的。不想自动续的会话，可在 GUI 把权限模式设为「卡住通知我」或把状态标为「暂停」——开机后只恢复上下文、不自动发指令，等手动点继续。

## 4. 数据存储与导出/导入迁移

### 本地存储（`%APPDATA%\claude-guard\`）
- `registry.db`（SQLite）：会话账本——目录、Claude 会话 ID、目标任务、权限模式、状态、轮次、时间戳。
- `config.json`：全局配置——默认权限模式、自启开关、静止检测秒数阈值、最大自动轮次等。
- `logs/`：每个会话的运行日志（PtyHost 抓到的输出留存，便于回看排错）。

### 上下文归属
Claude 完整对话上下文**不归本项目存**，在 `~/.claude/projects/<项目哈希>/<会话ID>.jsonl`。账本只是「指针」，记着会话 ID 对应哪个任务。不复制、不解析 Claude 内部格式，只引用。

### 导出（A 机）
GUI 点「导出」，Snapshotter 生成 zip，含：
1. `registry.db` + `config.json`
2. 被选中会话对应的 `.jsonl` 上下文文件（从 `~/.claude/projects/` 捞出）
3. 清单文件（每个会话的原始工作目录路径、导出时间、版本号）

可选「导出全部」或「只导出某几个」。

### 导入（B 机）
B 机装了 claude-guard，点「导入」选 zip：
1. 把 `.jsonl` 原进 B 机 `~/.claude/projects/` 对应位置。
2. 会话记录并入 B 机 `registry.db`。
3. **处理路径差异**（迁移最大的坑）：原路径在 B 机不存在时，GUI 弹窗让用户**重新指定该项目在本机的目录**，把会话工作目录改成新路径。
4. 还原后这些会话在 B 机为「暂停」状态，点继续（或开了自启自动续）即接着 A 机上下文往下跑。

### 边界
跨机能跑的前提：两台机器都装了 Claude Code、都登录了可用账号、相关项目代码都在（或一并拷过去）。本项目迁移「会话上下文 + 任务账本」，不负责迁移项目源代码和 Claude 登录凭证（代码本应在 git 里）。

## 5. 错误处理与异常边界

核心思路：**宁可停下来等用户、也不要盲目自动操作**。

1. **静止检测误判**：N 秒阈值 GUI 可配（默认保守值如 60 秒）；发下一条前还需确认 Claude 真的回到「等待输入」提示符状态，两个条件都满足才算一轮结束。
2. **续跑死循环/跑飞**：每会话设**最大自动轮次上限**（GUI 可配，默认如 20 轮），到顶暂停并在 GUI 标红等介入；每轮记日志便于回看。
3. **Claude 进程意外退出/崩溃**：PtyHost 监控子进程。异常退出时**不盲目重启**，标为「异常中断」，记录退出码和最后输出，GUI 提示；用户可一键「重新恢复」或手动查看。避免崩溃-重启死循环。
4. **会话 ID 失效 / `.jsonl` 缺失**：恢复前校验 `.jsonl` 是否存在，不存在则明确报「上下文文件缺失，无法恢复」而非静默失败；导入时同样校验。
5. **权限询问识别**：靠匹配输出特征识别。识别规则做成**配置文件里可更新的模式**，而非写死代码；识别不确定时保守处理——当成「卡住」通知用户，不乱按。
6. **多会话并发**：每会话独立 PtyHost 进程和日志，互不干扰；SQLite 写入加锁避免账本损坏。
7. **数据损坏兜底**：SQLite 账本每次重要变更后保留上一版本备份；启动时检测到主库损坏则自动回退备份并提示。

## 6. 测试策略

核心原则：引擎与 GUI 解耦，引擎可脱离界面被完整测试。

1. **单元测试**
   - SessionRegistry：临时 SQLite 测增删改查、状态流转、并发写锁、损坏回退。
   - IdleDetector：喂预录输出流样本（正常输出、卡权限询问、长静默），断言判断正确，不需真 Claude。
   - Snapshotter：造假 registry + 假 `.jsonl`，测导出 zip 内容、导入还原、路径差异处理，全在临时目录。
2. **PtyHost 测试（最关键）**：写一个**假的交互式程序**（小 Python 脚本，模拟 Claude：打印、停下等输入、模拟权限询问），让 PtyHost 托管它，反复快速确定性地验证「读输出、写指令、识别静止、识别权限询问」全套，不依赖真 Claude。
3. **Supervisor 集成测试**：PtyHost（接假 Claude）+ IdleDetector + Registry 组合，跑完整「拉起→检测静止→自动应答→发下一条→更新状态→到轮次上限暂停」流程，断言状态机正确。
4. **真 Claude 冒烟测试**：少量手动端到端——真托管会话、真关重开恢复、真导出导入。
5. **GUI**：逻辑薄，手动验收为主，不强求自动化 UI 测试（YAGNI）。

测试金字塔的底座是「假 Claude 程序」——让最危险的 PTY 交互能被确定性反复测试。

## 7. 技术选型小结

- 语言：Python 3
- GUI：PySide6（可打包 exe）
- PTY：`pywinpty`（Windows）；设计上预留跨平台抽象（Linux/macOS 用标准 `pty`）
- 存储：SQLite（标准库 `sqlite3`）
- 打包：后续选 PyInstaller 等（实现阶段定）
